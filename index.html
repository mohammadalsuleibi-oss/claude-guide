Claude Code — Team Deep Dive Guide

Source: code.claude.com/docs May 2026

Feature Landscape Map

Product Definition

Claude Code is an agentic coding tool — not a chat assistant. It reads your codebase, edits files, runs commands, and orchestrates multi-step work across your entire project. It operates on your local machine with your tools, your filesystem, and your credentials.

Surface Layer — Where It Runs

Terminal CLIclaudeFull feature set, pipe/stdin, scriptingSubscription or API key
VS Code extensionMarketplaceInline diffs, @-mentions, plan reviewCLI installed
JetBrains pluginMarketplaceInteractive diffs, selection contextCLI installed

Web (claude.ai/code)BrowserNo local setup, long-running cloud tasks, iOS appPaid subscription
Remote ControlCLI / menu or mobileContinue local session from phone/browserActive local session
Critical distinction

All surfaces share the same underlying engine — CLAUDE.md, settings, MCP servers, and skills work across all of them.

Feature Deep Dives

Memory & Context

Source: code.claude.com/docs/en/memory

The Core Problem

Every Claude Code session starts with a fresh context window. Nothing persists by default. Memory is the mechanism to bridge sessions.

Two Complementary Systems

CLAUDE.md filesAuto Memory
Who writes itYou (human)Claude (automatic)
What it containsInstructions, rules, standardsLearnings, build commands, preferences Claude discovers
ScopeProject / User / Org / LocalPer repository, shared across worktrees
Loaded into contextEvery session, full contentEvery session, first 200 lines or 25KB of MEMORY.md
Use forCoding standards, workflows, architecture decisionsBuild commands, debugging insights, patterns Claude discovers
LocationVarious (see scopes below)~/.claude/projects/<project>/memory/
Key rule

CLAUDE.md = what you want Claude to always know. Auto memory = what Claude learns from watching you work.

CLAUDE.md — Scope Hierarchy

paths
/Library/Application Support/ClaudeCode/CLAUDE.md   ← Managed (org-wide, cannot be excluded)
~/.claude/CLAUDE.md                                  ← User (all your projects)
./CLAUDE.md or ./.claude/CLAUDE.md                  ← Project (team-shared via git)
./CLAUDE.local.md                                    ← Local personal (gitignored)

Load order: Managed → User → Project → Local. More specific = higher priority. All discovered files are concatenated, not overridden.

Subdirectory CLAUDE.md files load lazily — only when Claude reads files in that subdirectory. They are NOT re-injected after /compact.

.claude/rules/ — Modular Instructions

For larger projects, break CLAUDE.md into topic files:

tree
.claude/
├── CLAUDE.md           ← main project instructions
└── rules/
    ├── code-style.md   ← always loaded
    ├── testing.md      ← always loaded
    └── api-design.md   ← always loaded (or path-scoped)

Path-scoped rules — only load when Claude works with matching files:

yaml
---
paths:
  - "src/api/**/*.ts"
---
# API Development Rules
- All API endpoints must include input validation

Use path-scoped rules when: instructions apply to one layer of the stack (e.g., Terraform files, Lambda handlers, test files).

Auto Memory

  • Stored at: ~/.claude/projects/<project>/memory/MEMORY.md + topic files
  • First 200 lines / 25KB of MEMORY.md loaded at every session start
  • Topic files (debugging.md, patterns.md) loaded on-demand by Claude
  • Machine-local — does NOT sync to cloud environments
  • Run /memory to browse, edit, or delete
  • Disable: CLAUDE_CODE_DISABLE_AUTO_MEMORY=1 or "autoMemoryEnabled": false

Subagents can maintain their own auto memory — see subagent config docs.

Context Management: /compact

When the context window fills up:

  • /compact replaces conversation history with a summary
  • Project-root CLAUDE.md survives compaction (re-read from disk and re-injected)
  • Nested CLAUDE.md files in subdirectories are NOT re-injected after compaction
  • Skills are re-attached within a combined 25K token budget (most recently invoked first)
  • Auto memory survives compaction
Production gotcha

Don't rely on in-conversation instructions for long tasks — always write them to CLAUDE.md before starting. Anything only said in chat will be lost after compaction.

CLAUDE.md Best Practices

  • Target under 200 lines per file. Longer = less adherence + more context cost.
  • Write verifiable, specific instructions: "Run npm test before committing" not "Test your changes"
  • Use markdown headers and bullets — Claude scans structure like humans do
  • If content is procedural (multi-step), move it to a Skill instead
  • If content is path-specific, use path-scoped rules instead
  • Use HTML block comments <!-- notes --> for maintainer notes — stripped from context
The test

If an entry in CLAUDE.md reads like a procedure, it belongs in a Skill. If it reads like a fact, it belongs in CLAUDE.md.

CLAUDE.md vs AGENTS.md

If your repo already uses AGENTS.md for other coding agents:

markdown
<!-- CLAUDE.md -->
@AGENTS.md

## Claude Code-specific
Use plan mode for changes under `src/billing/`.

Or symlink: ln -s AGENTS.md CLAUDE.md

Sub-agents

Source: code.claude.com/docs/en/sub-agents

What a Sub-agent Is

A sub-agent is a separate Claude instance with its own context window, tool allow-list, and system prompt. The main session delegates a task to it, the sub-agent works in isolation, and only the summarized result returns to the parent. Your main conversation stays clean.

Two ways to invoke:

  • Auto-delegated — Claude reads the sub-agent's description and decides to use it
  • Manual — @agent-name from the user

Two ways to define:

  • Built-in types — Explore (read-only research), Plan (planning), general-purpose (anything)
  • Custom — markdown files in .claude/agents/ (project) or ~/.claude/agents/ (personal)

Anatomy of a Custom Sub-agent

path
.claude/agents/code-reviewer.md
markdown
---
name: code-reviewer
description: Reviews code for quality and best practices. Use proactively after code changes.
tools: Read, Grep, Glob, Bash(git diff *)
model: sonnet
permissionMode: plan
maxTurns: 20
skills:
  - api-conventions
  - error-handling-patterns
memory: project
---

You are a senior code reviewer. When invoked:
1. Run `git diff` to see recent changes
2. Focus on modified files
3. Review for readability, error handling, security, test coverage

Organize feedback by priority: Critical → Warnings → Suggestions.
Key frontmatter fields
FieldPurpose
descriptionDrives auto-delegation. Include "use proactively" / "use when X" trigger phrases.
toolsComma-separated allow-list. Omit to inherit all. Supports Agent(agent_type) syntax.
disallowedToolsSubtractive deny-list. Use Skill(skill-name) to block specific skills.
modelhaiku, sonnet, opus, or inherit (default).
permissionModedefault, acceptEdits, plan, bypassPermissions.
maxTurnsHard cap on agentic turns.
skillsSkill names to preload into the sub-agent's startup context (see Section 3).
memoryuser, project, or local — persistent memory scope for this sub-agent.
backgroundtrue runs as a background task.

Isolation Rules

PropertyBehavior
Conversation historySub-agent does NOT inherit parent history. Fresh context.
CLAUDE.mdLoaded fresh (managed + user + project), like any session.
Auto memorySub-agent has its own scope per memory: field.
Skills (descriptions)NOT auto-injected into sub-agent system prompt — must use skills: field for preloading.
Skills (filesystem)Sub-agents with file access can still discover skills in .claude/skills/ via Glob and invoke them. Blocking requires disallowedTools: Skill(name).
ToolsRestricted by tools / disallowedTools.
ResultSummary returned to parent — full sub-agent context discarded.
Practical implication

The skills: field controls startup injection (full skill content in system prompt), not access (filesystem discovery). If you need a sub-agent locked down, combine tools: allow-list with disallowedTools: Skill(...).

Parallelism

  • Sub-agents can be chained (one after another) or run in parallel from the parent
  • The parent issues multiple Agent tool calls; each runs in its own process
  • Hard constraint: sub-agents cannot spawn other sub-agents. Plan chains accordingly — your delegation tree is one level deep.
  • Each sub-agent's messages include a parent_tool_use_id for observability across parallel runs

Lifecycle

Sub-agent lifecycle: spawn → run → return summary → discard context

When to Use Sub-agents

Use caseWhy
Heavy research / code archaeologyKeep noise out of main context; use Explore agent
Parallel workSpawn N sub-agents simultaneously — N independent contexts
Specialized domain workCustom agent with preloaded skills: for that domain
Risk isolationRestricted tools: and permissionMode: plan
Long file-reading tasksParent stays clean; only summary returns

Anti-patterns: Don't use a sub-agent for a task that needs ongoing conversation with the user (no history inheritance). Don't use one when you need the parent to see intermediate reasoning (only the summary returns).

Sub-agent Troubleshooting

Skills

Source: code.claude.com/docs/en/skills

What Skills Are

Skills are reusable, named workflows defined as Markdown files. They solve the problem of repeating the same multi-step instructions in chat, and the problem of CLAUDE.md growing into a procedure manual.

The key distinction from CLAUDE.md:

  • CLAUDE.md content → always in context, every token costs every session
  • Skill content → only loads when invoked (or when description matches the prompt)

The key distinction from Hooks:

  • Hooks → deterministic shell triggers at lifecycle events (runs regardless of Claude's decisions)
  • Skills → Claude-orchestrated workflows (Claude decides when to run them, unless you restrict)

Anatomy of a Skill

tree
~/.claude/skills/my-skill/        ← personal scope
├── SKILL.md                      ← required: instructions + frontmatter
├── reference.md                  ← optional: detailed docs, loaded on demand
├── examples/
│   └── sample.md
└── scripts/
    └── helper.py                 ← scripts Claude can execute
markdown
---
name: my-skill
description: What this skill does and when to use it. Put the KEY USE CASE first.
disable-model-invocation: true    ← only you can invoke (not Claude auto)
context: fork                     ← run in isolated subagent context
agent: Explore                    ← which subagent type
allowed-tools: Bash(git add *) Bash(git commit *)
effort: high
paths:
  - "src/**/*.ts"                 ← only activate for TypeScript files
---

Your instructions here...
Note

Custom commands (.claude/commands/deploy.md) have been merged into skills. Both paths work. Skills add: supporting files directory, frontmatter control, subagent execution. Skills take precedence if names conflict.

Skill Scopes

ScopePathVisibility
Enterprise (managed)/Library/.../ClaudeCode/skills/All org users — overrides all
Personal~/.claude/skills/<name>/SKILL.mdAll your projects
Project.claude/skills/<name>/SKILL.mdThis project, team-shared via git
Plugin<plugin>/skills/<name>/SKILL.mdWhere plugin is enabled, namespaced

Precedence: Enterprise > Personal > Project. Plugin skills use plugin-name:skill-name namespace — no conflicts.

Invocation Control

FrontmatterYou can invokeClaude can invokeDescription in context
(default)✅✅Always included
disable-model-invocation: true✅❌NOT in context
user-invocable: false❌✅Always included

Rule of thumb:

  • Side-effect skills (/deploy, /commit, /send-slack-message) → always disable-model-invocation: true. You don't want Claude deciding to deploy because your code "looks ready."
  • Background knowledge skills (legacy-system-context) → user-invocable: false. Claude should load it when relevant; it's not a user command.

Dynamic Context Injection

Skills can execute shell commands before Claude sees the content:

markdown
---
name: summarize-changes
description: Summarizes uncommitted changes and flags anything risky.
---

## Current changes
!`git diff HEAD`

## Instructions
Summarize the above in 2-3 bullets. Flag missing error handling, hardcoded values, or tests that need updating.

The !`git diff HEAD` line runs at skill load time. Claude receives the rendered output, not the command. This is preprocessing, not something Claude executes.

Multi-line form:

markdown
## Environment
```!
node --version
npm --version
git status --short
```

To disable shell execution for security: set "disableSkillShellExecution": true in managed settings. Replaces all ! blocks with [shell command execution disabled by policy].

Running Skills in a Subagent (context: fork)

markdown
---
name: pr-summary
description: Summarize changes in a pull request
context: fork
agent: Explore
allowed-tools: Bash(gh *)
---

## Pull request context
- PR diff: !`gh pr diff`
- PR comments: !`gh pr view --comments`

## Your task
Summarize this pull request...

When context: fork:

  • Skill content becomes the subagent prompt
  • Subagent runs in isolation — no access to your conversation history
  • agent field picks the execution environment (Explore, Plan, general-purpose, or any custom subagent in .claude/agents/)
  • Results are summarized and returned to your main session

Use context: fork when: the skill is a bounded research or analysis task that should not touch your main conversation state.

Skill Content Lifecycle & Token Costs

  • Once invoked, skill content enters the conversation as a message and stays for the session
  • Claude does NOT re-read the skill file on later turns
  • After /compact: the top 5,000 tokens of each recently-invoked skill are re-attached
  • Combined re-attach budget: 25,000 tokens across all skills
  • Older skills get dropped first if budget is exceeded
Production implication

Keep SKILL.md under 500 lines. Move detailed reference docs to supporting files. A skill that's too large will silently lose its tail after compaction.

Pre-approving Tools in a Skill

markdown
---
name: commit
description: Stage and commit the current changes
disable-model-invocation: true
allowed-tools: Bash(git add *) Bash(git commit *) Bash(git status *)
---
Security note

allowed-tools in a project skill takes effect after you accept the workspace trust dialog for that folder. Review project skills before trusting a repository — a skill can grant itself broad tool access.

The default. Full feature set: hooks, skills, MCP, auto memory, permission prompts, diff review.

claude

Headless Mode (-p)

bash
# Basic
claude -p "What does the auth module do?"

# Pipe stdin
cat build-error.txt | claude -p 'explain root cause' > output.txt

# Structured JSON output
claude -p "Summarize this project" --output-format json

# Tool restriction
claude -p "Run the test suite and fix any failures" \
  --allowedTools "Bash,Read,Edit"

# Session continuity
session_id=$(claude -p "Start a review" --output-format json | jq -r '.session_id')
claude -p "Continue that review" --resume "$session_id"

Output formats: text (default) | json | stream-json

Bare Mode (--bare) — Critical for CI

bash
claude --bare -p "Summarize this file" --allowedTools "Read"

--bare skips everything auto-discovered: hooks, skills, plugins, MCP servers, auto memory, CLAUDE.md. Result is deterministic regardless of local developer config.

Use --bare for all CI/CD pipelines

Without it, a teammate's local ~/.claude hook or MCP server could silently affect your pipeline run.

Note: --bare will become the default for -p in a future release. Start using it now.

In bare mode, authentication must be ANTHROPIC_API_KEY or apiKeyHelper — no OAuth.

stdin cap: As of v2.1.128, piped stdin is capped at 10MB. For larger inputs, write to a file and reference the path in the prompt.

CI/CD Integration Pattern

yaml
# GitHub Actions example
- name: Claude Code review
  run: |
    gh pr diff "$PR_NUMBER" | claude --bare -p \
      --append-system-prompt "You are a security engineer. Review for vulnerabilities." \
      --output-format json \
      --allowedTools "Read"

Permission hygiene for CI:

  • Always use --allowedTools to explicitly scope tool access
  • Prefer --permission-mode dontAsk for locked-down runs
  • Use --output-format json to capture cost metadata (total_cost_usd)
  • Never run with bypassPermissions in CI

The Two Directions: Skills ↔ Sub-agents

Skills and sub-agents look similar but solve inverse problems. The same primitive — a skill's context: fork field, and a sub-agent's skills: field — can compose in either direction.

The Claude Code docs describe both: a skill can fork into a sub-agent, and a sub-agent can preload skills at startup. They use the same underlying mechanism but inject content into different sides of the boundary.

Direction A — Skill → Sub-agent (context: fork in SKILL.md)

The skill IS the task. The sub-agent is just the execution environment.

yaml — .claude/skills/terraform-drift/SKILL.md
---
name: terraform-drift
description: Check for drift between state and actual infra
context: fork
agent: Explore                  # which built-in or custom subagent runs this
allowed-tools: Bash(terraform *)
disable-model-invocation: true  # you invoke it, not Claude auto
---

## Live state
!`terraform plan -no-color 2>&1`

## Your task
Identify any drift between the plan output above and the expected state.
Flag any resources being replaced, destroyed, or modified outside the
module boundary. Return a structured summary only.
  • The skill content becomes the prompt that drives the sub-agent
  • The ! injection runs before Claude sees anything — the sub-agent receives a fully-rendered prompt with live data already inlined
  • The result is summarized and returned to your main session
  • Your main conversation context stays clean

Use this when: the workflow is a bounded, triggered task — a scan, a summary, a review. You define the task once as a skill; the sub-agent is disposable.

Direction B — Sub-agent → Skills (skills: in agent frontmatter)

The skills are domain knowledge. The sub-agent is the persistent specialist.

yaml — .claude/agents/terraform-developer.md
---
name: terraform-developer
description: Writes and reviews Terraform following team standards. Use proactively for any IaC work.
tools: Read, Edit, Write, Bash(terraform *), Bash(aws *)
model: sonnet
skills:
  - aws-naming-conventions      # full content injected at startup
  - terraform-module-standards  # full content injected at startup
  - iam-policy-patterns         # full content injected at startup
memory: project
---

You are a Terraform specialist. Apply the preloaded conventions and patterns
to all IaC work. Update memory with project-specific decisions as you discover them.
  • The full content of each listed skill is injected into the sub-agent's context at startup
  • The sub-agent carries this knowledge for its entire lifetime
  • Unlike a regular session where skill descriptions are loaded but full content only on invocation, here full content is in context from turn one

Use this when: you have a reusable specialist that needs standing domain knowledge before it sees any task — conventions, standards, patterns, architectural decisions.

The Combined Pattern

Two mechanisms in one flow: solid arrow = preloading via skills: · dashed arrow = runtime invocation via filesystem discovery
Two distinct mechanisms in play

1. Preloading (skills: field) — full skill content is injected into the sub-agent's system prompt at startup. Standing knowledge, always present.

2. Filesystem discovery — at runtime, any sub-agent with file-access tools (Read, Glob) can scan .claude/skills/ and invoke skills it discovers. This is how a sub-agent reaches a skill that wasn't preloaded.

In the diagram: aws-naming-conventions, iam-policy-patterns, and terraform-module-standards are preloaded (always in context). /terraform-drift is discovered and invoked at runtime — it's not in the skills: list, but the sub-agent's Read/Glob tools let it find and call any skill in .claude/skills/.

Want to actually block runtime discovery?

Don't rely on omitting from skills: — that only affects preloading. Use disallowedTools: Skill(skill-name) in the sub-agent frontmatter. See "Critical Technical Differences" table above.

The sub-agent provides persistent specialist identity. The skill provides isolated task execution. They compose because they solve different scoping problems — one is about what a worker knows, the other is about what a task does.

Hard constraint

Sub-agents cannot spawn other sub-agents. So the forked sub-agent from a context: fork skill cannot itself spawn another. Keep this in mind when designing chains — the delegation tree is one level deep.

Quick mental model
  • Slash commands = reusable prompts for single-shot tasks. Best for standardizing PRs, running tests, simple refactors.
  • Skills = on-demand knowledge and deterministic workflows with progressive disclosure (description loaded; full content only on invocation). Best for API docs, style guides, multi-step file manipulations.
  • Sub-agents = isolated workers with their own context window, for complex multi-step work that shouldn't pollute the main conversation. Best for TDD workflows, security audits, root cause analysis, parallel execution.

  • Verifiable, specific instructions only
  • Use path-scoped rules for per-filetype standards
  • Version-control project CLAUDE.md; gitignore CLAUDE.local.md
  • Run /init with CLAUDE_CODE_NEW_INIT=1 for interactive setup

Sub-agents

  • Always include trigger phrases in description for auto-delegation ("use proactively", "use when X")
  • Restrict tools: to the minimum the sub-agent needs
  • Use permissionMode: plan for any agent that shouldn't make changes (e.g., reviewers, explorers)
  • Combine tools: allow-list with disallowedTools: Skill(name) to actually block specific skills
  • Use skills: for standing domain knowledge a specialist needs from turn one
  • Remember the one-level-deep constraint — design chains, not trees

Skills

  • Use disable-model-invocation: true for any skill with side effects
  • Keep SKILL.md under 500 lines; move reference docs to supporting files
  • Use ! shell injection to pull live data into the prompt (git diff, env info)
  • Use context: fork to isolate skills from main conversation state
  • Review project skills before accepting workspace trust — they can grant broad tool access
  • Set "disableSkillShellExecution": true in managed settings for org security

Execution Modes / Headless

  • Always --bare for CI/CD pipelines
  • Always --allowedTools with explicit tool list for automated runs
  • Use --output-format json to capture cost metadata
  • Pipe stdin capped at 10MB — write large inputs to file
  • Use ANTHROPIC_API_KEY (not OAuth) in all non-interactive contexts

Permission Hygiene

  • bypassPermissions → local sandbox only, never prod
  • acceptEdits → CI where you want file writes without prompts but still want shell command control
  • dontAsk → locked-down CI where unknown tools should fail the run
  • Always use --allowedTools with explicit pattern matching: Bash(git diff *) not Bash
